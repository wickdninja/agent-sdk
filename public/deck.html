Why Ephemeral Tokens?

üîê Security Benefits

  • No API key exposure - Keys stay server-side only
  • Short-lived access - Tokens have a short (TTL)
  • Per-session auth - Each session gets unique token
  • Rate limiting - Control requests per user/IP
  • Audit trail - Track usage per session/user

‚ùå Without Tokens (Server Agent)

  • All audio must proxy through server
  • Higher latency (2x network hops)
  • Increased server costs (bandwidth)
  • Complex WebSocket state management
  • Scaling requires media servers

Client WebRTC Implementation: Step by Step

agent.js - Annotated Connection Flow 
class RealtimeVoiceAgent {
  async connect() {
    // Step 1: Get ephemeral token (security layer)
    const tokenResponse = await fetch('/api/session');
    const data = await tokenResponse.json();
    
    // Step 2: Create the main connection object (like opening a phone line)
    this.pc = new RTCPeerConnection({
      iceServers: [{ urls: 'stun:stun.l.google.com:19302' }]
      // STUN server helps find your public IP behind router
    });
    
    // Step 3: Connect your microphone (with quality improvements)
    const stream = await navigator.mediaDevices.getUserMedia({ 
      audio: {
        echoCancellation: true,  // Prevents feedback loops
        noiseSuppression: true,  // Removes background noise
        autoGainControl: true    // Normalizes volume
      }
    });
    stream.getTracks().forEach(track => 
      this.pc.addTrack(track, stream)
    );
    
    // Step 4: Create data channel (for non-audio communication)
    this.dc = this.pc.createDataChannel('oai-events', {
      ordered: true  // Messages arrive in order sent
    });
    
    // Step 5: Negotiate connection (WebRTC handshake)
    const offer = await this.pc.createOffer();  // "I want to connect"
    await this.pc.setLocalDescription(offer);    // Save our offer
    await this.pc.setRemoteDescription(data.answer); // Accept OpenAI's response
  }
}

üéØ Key Point: Each component has a specific job - peer connection for audio, data channel for everything else

Understanding WebRTC: The Two Channels

Why We Need Two Different Channels

üéôÔ∏è Audio Channel (RTCPeerConnection)

Purpose: Real-time voice streaming

  • Your voice ‚Üí OpenAI
  • AI voice ‚Üí Your speakers
  • Continuous streaming (like a phone call)
  • Handles network complexity (NAT/firewall)

Think of it as: The phone line

üì® Data Channel

Purpose: Everything else

  • AI configuration & instructions
  • Tool execution requests
  • Live transcripts (both sides)
  • Interruption detection events
  • Session state management

Think of it as: Text messages alongside the call

‚ùì Common Misconception

"Can't we just send everything through audio?" No! Here's why:

  • üö´ Audio is for continuous streaming, not structured data
  • üö´ Tool calls need JSON format, not speech
  • üö´ Transcripts would create feedback loops if sent as audio
  • ‚úÖ DataChannel provides reliable, ordered message delivery

Voice Activity Detection (VAD): The Secret to Natural Conversation

What is VAD?

VAD determines when someone has finished speaking so the agent knows when to respond.

Three VAD Options 
// Option 1: Server VAD (Our choice)
turn_detection: {
  type: 'server_vad',
  threshold: 0.5,
  silence_duration_ms: 200
}

// Option 2: No VAD (Push-to-talk)
turn_detection: { type: 'none' }
// User must manually signal done talking

// Option 3: Client VAD (You detect)
// You implement VAD and send signals

Why VAD Settings Matter

threshold: 0.5

  • üìä Lower (0.3): Triggers on background noise
  • üìä Higher (0.8): Might miss soft speech
  • ‚úÖ 0.5: Good balance for most environments

silence_duration_ms: 200

  • ‚è±Ô∏è 100ms: Too quick, cuts off mid-thought
  • ‚è±Ô∏è 500ms: Feels sluggish, unnatural pauses
  • ‚úÖ 200ms: Natural conversation flow

Real Coffee Shop Impact

"I'll have a... um... large coffee"

  • ‚ùå Bad VAD: Interrupts at "um"
  • ‚úÖ Good VAD: Waits for complete order

Core Agent Functions: Tool Execution Architecture

agent.js - executeFunction() Implementation 
async executeFunction(name, args) {
  console.log(`[Tool] Executing: ${name}`, args);
  
  // Map tool names to API endpoints
  const endpoints = {
    'fetch_menu': '/api/menu',
    'set_user': '/api/user',
    'confirm_order': '/api/order/confirm',
    'submit_order': '/api/order/submit',
    'get_order_history': '/api/order/history'
  };
  
  try {
    // 1. Call our backend API (not OpenAI)
    const response = await fetch(endpoints[name], {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ ...args, sessionId: this.sessionId })
    });
    
    const result = await response.json();
    
    // 2. Update local state based on tool results
    if (name === 'set_user') {
      this.currentUserId = result.userId;
      this.updateOrderStatus(`Welcome, ${result.name}!`);
    }
    
    // 3. Send result back to OpenAI through DataChannel
    this.dc.send(JSON.stringify({
      type: 'conversation.item.create',
      item: {
        type: 'function_call_output',
        call_id: this.lastCallId,  // Links result to request
        output: JSON.stringify(result)
      }
    }));
    
    // 4. Trigger agent to respond with the result
    this.dc.send(JSON.stringify({ 
      type: 'response.create'  // "Now speak about this result"
    }));
    
  } catch (error) {
    console.error(`[Tool] Error executing ${name}:`, error);
    // Send error back to agent
    this.sendToolError(error.message);
  }
}

Message Handling: The Event-Driven Architecture

agent.js - Core Message Types 
// Different message types and what they mean for our app
this.dc.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  
  switch(msg.type) {
    // === CONVERSATION EVENTS ===
    case 'conversation.item.created':
      // New message added (user or assistant)
      if (msg.item.role === 'user') {
        this.addMessage('user', msg.item.content);
      }
      break;
    
    // === AUDIO EVENTS ===
    case 'input_audio_buffer.speech_started':
      // User started speaking
      this.showUserSpeaking();
      break;
      
    case 'input_audio_buffer.speech_stopped':
      // User stopped (VAD triggered)
      this.showProcessing();
      break;
    
    // === RESPONSE EVENTS ===
    case 'response.audio.transcript.delta':
      // Incremental speech from agent
      this.currentTranscript += msg.delta;
      this.updateAgentMessage(this.currentTranscript);
      break;
      
    case 'response.audio.transcript.done':
      // Agent finished speaking
      this.finalizeAgentMessage();
      break;
    
    // === TOOL EVENTS ===
    case 'response.function_call_arguments.start':
      // Tool call starting
      this.pendingToolCall = { name: msg.name, arguments: '' };
      break;
      
    case 'response.function_call_arguments.delta':
      // Building tool arguments
      this.pendingToolCall.arguments += msg.delta;
      break;
      
    case 'response.function_call_arguments.done':
      // Execute the tool
      const args = JSON.parse(this.pendingToolCall.arguments);
      this.executeFunction(this.pendingToolCall.name, args);
      break;
    
    // === ERROR EVENTS ===
    case 'error':
      console.error('[Agent Error]', msg.error);
      this.handleError(msg.error);
      break;
  }
};

Key Insight: Every user action and agent response flows through these events - it's the nervous system of our voice agent!

Network Traversal: STUN vs TURN

Essential WebRTC infrastructure - Why your calls sometimes fail

üîç STUN - Session Traversal Utilities for NAT

  • Purpose: Discovers your public IP address
  • How: "What's my IP from the internet's perspective?"
  • Cost: Free (like asking for directions)
  • Success rate: ~80% of connections
  • Fails when: Symmetric NAT, strict firewalls

üîÑ TURN - Traversal Using Relays around NAT

  • Purpose: Relays all traffic when direct fails
  • How: "I'll forward everything between you"
  • Cost: $$$ (bandwidth both directions)
  • Success rate: ~99.9% (always works)
  • Trade-off: +20-50ms latency, ongoing costs

Real-World Analogy üè¢

STUN = Finding your office building's address
TURN = Using a courier service when direct delivery fails

The Real-World Analogy üè†

STUN = Looking in a mirror

You're behind your home router (NAT). STUN is like calling a friend and asking "What number shows up on your caller ID?" Now you know your public phone number!

TURN = Using a conference call service

When direct calling fails (blocked by firewall), you both dial into a conference bridge. More expensive, slight delay, but always works!

üçï Fun Fact: The Pizza Delivery Problem

STUN/TURN is like ordering pizza to an office building:

  • STUN: Tells you the building's address (555 Main St) but not your suite (Suite 401)
  • Router: The receptionist who knows to send "Suite 401" pizzas to you
  • TURN: When security won't let delivery in, someone meets them in the lobby and brings it up

Managing Session State

services/session-manager.js 
class SessionManager {
  constructor() {
    this.sessions = new Map();
    this.startCleanupInterval();
  }
  
  getOrCreateSession(sessionId) {
    if (!this.sessions.has(sessionId)) {
      this.sessions.set(sessionId, {
        userId: null,
        currentOrder: { items: [] },
        lastActivity: Date.now(),
        context: {}
      });
    }
    return this.sessions.get(sessionId);
  }
  
  startCleanupInterval() {
    setInterval(() => {
      const now = Date.now();
      for (const [id, session] of this.sessions) {
        if (now - session.lastActivity > 30 * 60 * 1000) {
          this.sessions.delete(id); // Clean after 30 min
        }
      }
    }, 60000); // Check every minute
  }
}
