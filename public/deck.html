Why Ephemeral Tokens?

üîê Security Benefits

  • No API key exposure - Keys stay server-side only
  • Short-lived access - Tokens have a short (TTL)
  • Per-session auth - Each session gets unique token
  • Rate limiting - Control requests per user/IP
  • Audit trail - Track usage per session/user

‚ùå Without Tokens (Server Agent)

  • All audio must proxy through server
  • Higher latency (2x network hops)
  • Increased server costs (bandwidth)
  • Complex WebSocket state management
  • Scaling requires media servers

Client WebRTC Implementation: Step by Step

agent.js - Annotated Connection Flow 
class RealtimeVoiceAgent {
  async connect() {
    // Step 1: Get ephemeral token (security layer)
    const tokenResponse = await fetch('/api/session');
    const data = await tokenResponse.json();
    
    // Step 2: Create the main connection object (like opening a phone line)
    this.pc = new RTCPeerConnection({
      iceServers: [{ urls: 'stun:stun.l.google.com:19302' }]
      // STUN server helps find your public IP behind router
    });
    
    // Step 3: Connect your microphone (with quality improvements)
    const stream = await navigator.mediaDevices.getUserMedia({ 
      audio: {
        echoCancellation: true,  // Prevents feedback loops
        noiseSuppression: true,  // Removes background noise
        autoGainControl: true    // Normalizes volume
      }
    });
    stream.getTracks().forEach(track => 
      this.pc.addTrack(track, stream)
    );
    
    // Step 4: Create data channel (for non-audio communication)
    this.dc = this.pc.createDataChannel('oai-events', {
      ordered: true  // Messages arrive in order sent
    });
    
    // Step 5: Negotiate connection (WebRTC handshake)
    const offer = await this.pc.createOffer();  // "I want to connect"
    await this.pc.setLocalDescription(offer);    // Save our offer
    await this.pc.setRemoteDescription(data.answer); // Accept OpenAI's response
  }
}

üéØ Key Point: Each component has a specific job - peer connection for audio, data channel for everything else

Understanding WebRTC: The Two Channels

Why We Need Two Different Channels

üéôÔ∏è Audio Channel (RTCPeerConnection)

Purpose: Real-time voice streaming

  • Your voice ‚Üí OpenAI
  • AI voice ‚Üí Your speakers
  • Continuous streaming (like a phone call)
  • Handles network complexity (NAT/firewall)

Think of it as: The phone line

üì® Data Channel

Purpose: Everything else

  • AI configuration & instructions
  • Tool execution requests
  • Live transcripts (both sides)
  • Interruption detection events
  • Session state management

Think of it as: Text messages alongside the call

‚ùì Common Misconception

"Can't we just send everything through audio?" No! Here's why:

  • üö´ Audio is for continuous streaming, not structured data
  • üö´ Tool calls need JSON format, not speech
  • üö´ Transcripts would create feedback loops if sent as audio
  • ‚úÖ DataChannel provides reliable, ordered message delivery

Data Channel: The Tool Execution Pipeline

agent.js - DataChannel Setup 
// Data channel handles all non-audio communication
this.dc.onopen = () => {
  // Send initial configuration
  this.dc.send(JSON.stringify({
    type: 'session.update',
    session: {
      tools: this.tools,
      instructions: `You are Bella, a friendly barista...`,
      turn_detection: {
        type: 'server_vad',
        threshold: 0.5,
        silence_duration_ms: 200
      }
    }
  }));
};

this.dc.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  
  switch(msg.type) {
    case 'response.audio.transcript.delta':
      // Agent is speaking
      this.updateTranscript(msg.delta);
      break;
      
    case 'response.function_call_arguments.done':
      // Tool call requested
      this.executeTool(msg.name, msg.arguments);
      break;
      
    case 'input_audio_buffer.speech_started':
      // User started speaking (interruption)
      this.handleInterruption();
      break;
  }
};

Key: DataChannel enables bidirectional event flow for tools and transcripts

Understanding STUN vs TURN

ü§ø Diving deeper - slightly off topic but fundamental to WebRTC (and why your video calls sometimes fail!)

üîç STUN - The Friendly Mirror

  • What it does: "Hey, what do I look like from the internet?"
  • Server replies: "You're at IP 73.162.x.x port 54321"
  • Cost: Free! (Like asking for directions)
  • Success rate: ~80% of connections
  • Fails when: Corporate firewalls, hotels, airports
// What we're using (free Google STUN)
iceServers: [{
  urls: 'stun:stun.l.google.com:19302'
}]

üîÑ TURN - The Expensive Middleman

  • What it does: "I'll relay everything for you"
  • Server says: "Send to me, I'll forward to them"
  • Cost: $$ (Pays for bandwidth both ways)
  • Success rate: ~99.9% (always works)
  • Downside: +20-50ms latency, costs money
// What we SHOULD use in production
iceServers: [{
  urls: 'turn:relay.xirsys.com:80',
  username: 'your-username',
  credential: 'your-password'
}]

The Real-World Analogy üè†

STUN = Looking in a mirror

You're behind your home router (NAT). STUN is like calling a friend and asking "What number shows up on your caller ID?" Now you know your public phone number!

TURN = Using a conference call service

When direct calling fails (blocked by firewall), you both dial into a conference bridge. More expensive, slight delay, but always works!

üçï Fun Fact: The Pizza Delivery Problem

STUN/TURN is like ordering pizza to an office building:

  • STUN: Tells you the building's address (555 Main St) but not your suite (Suite 401)
  • Router: The receptionist who knows to send "Suite 401" pizzas to you
  • TURN: When security won't let delivery in, someone meets them in the lobby and brings it up

Managing Session State

services/session-manager.js 
class SessionManager {
  constructor() {
    this.sessions = new Map();
    this.startCleanupInterval();
  }
  
  getOrCreateSession(sessionId) {
    if (!this.sessions.has(sessionId)) {
      this.sessions.set(sessionId, {
        userId: null,
        currentOrder: { items: [] },
        lastActivity: Date.now(),
        context: {}
      });
    }
    return this.sessions.get(sessionId);
  }
  
  startCleanupInterval() {
    setInterval(() => {
      const now = Date.now();
      for (const [id, session] of this.sessions) {
        if (now - session.lastActivity > 30 * 60 * 1000) {
          this.sessions.delete(id); // Clean after 30 min
        }
      }
    }, 60000); // Check every minute
  }
}
